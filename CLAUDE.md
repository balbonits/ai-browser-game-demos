# CLAUDE.md

Guidance for Claude Code working in this repo. See also [`AGENTS.md`](AGENTS.md) for tool-agnostic agent guidance and [`docs/`](docs/) for deeper details.

## What this repo is

A showcase of browser games and game demos built entirely by AI. Code is generated by Claude Code; pixel art is generated by PixelLab.ai (via MCP). The human's role is direction and review.

## Ground rules

- **No hand-written game code.** All game code should be produced by Claude (or another agent). If the user hand-writes something, they'll note it.
- **No build step, no dependencies.** Games are static HTML/CSS/JS. A `package.json` exists solely for the `npm run dev` convenience script — keep it dependency-free. No bundler; `npm install` must not be required for games to play.
- **Vanilla by default, engines when they fit.** Most games start as plain canvas + JS. **Phaser** and **PixiJS** are welcome on games where their features (scenes, physics, sprite batching, tweens) actually pay off — load them via CDN. Confirm the engine choice with the user the first time it's used on a given game.
- **Still off-limits without approval:** UI frameworks (React, Vue, Svelte), TypeScript, Tailwind, or anything that requires a build step.
- **Self-contained games.** Each game lives in `games/<slug>/` and owns its assets. Do not reach outside that folder for game-specific code or art.
- **Register new games** in `games.js` so they appear on the landing page.
- **Keep the landing page simple.** `index.html`, `games.js`, and `styles.css` at the root are shared infrastructure — change them only when adding/removing games or adjusting the shared shell.

## Adding a new game

See [`docs/adding-a-game.md`](docs/adding-a-game.md) for the step-by-step checklist. TL;DR:

1. Create `games/<slug>/index.html` (plus CSS/JS/assets as siblings).
2. Add an entry to `GAMES` in `games.js`.
3. Put any PixelLab-generated art under `games/<slug>/assets/`.

## Art pipeline (PixelLab MCP)

The PixelLab MCP is configured for this project. Available tool families:

- `create_character`, `animate_character`, `list_characters`, `get_character`, `delete_character`
- `create_topdown_tileset`, `create_sidescroller_tileset`, `create_tiles_pro`
- `create_isometric_tile`, `create_map_object`
- Plus `get_*`, `list_*`, `delete_*` variants

See [`docs/pixellab.md`](docs/pixellab.md) for workflow details (jobs are async — you get a job ID back and poll, download the resulting image, and save it under the game's `assets/` folder).

## Style

- Modern vanilla JS (ES modules fine if supported by the browser directly).
- Phaser / PixiJS allowed via CDN when the game earns it (see ground rules).
- Small, readable files — games here are demos, not shipping products.

## When in doubt

- Ask before introducing a build step, bundler, or `package.json`.
- Ask before introducing an engine (Phaser/PixiJS) for the first time on a given game.
- Ask before restructuring shared files (`index.html`, `games.js`, `styles.css`).
- Proceed freely inside a single `games/<slug>/` folder.

## Subagents

Two project-level subagents are defined under `.claude/agents/`:

- **`artist`** — PixelLab MCP asset generation and organization. Delegate anything involving sprites, animations, or tilesets.
- **`dev`** — vanilla-JS (or Phaser/PixiJS) game code. Delegate anything involving HTML/CSS/JS, game loops, collision, input, UI.

See [`docs/agents.md`](docs/agents.md) for when and how to use them.
